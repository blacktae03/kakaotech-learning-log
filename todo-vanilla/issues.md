## AI 활용 내역

### Todo CRUD 구현하기
- AI 활용 내용 : `{ id, text, completed }` 구조로 상태를 배열로 관리하고, 변경이 생길 때마다 `render()`를 호출하는 흐름을 직접 설계한 뒤, 이벤트 위임 방식으로 목록 버튼을 처리하는 부분의 코드 방향이 맞는지 AI에게 검토받았다.
- 직접 수정한 부분 : 추가 버튼 클릭 외에 Enter 키로도 추가할 수 있도록 이벤트를 추가했다. 또한 수정 모드를 `editingId` 상태 변수로 관리해 `render()` 안에서 해당 항목만 입력 폼으로 그리는 방식으로 직접 구조를 잡았다.
- 수정 이유 : DOM을 직접 바꾸는 방식으로 먼저 시도했더니 여러 항목을 동시에 수정 모드로 만들 수 있는 버그가 생겼다. 상태로 관리하면 `render()` 흐름이 일관되게 유지된다는 걸 직접 확인하고 바꿨다.

### 상태별 필터링 구현하기
- AI 활용 내용 : `currentFilter` 변수 하나로 탭 상태를 관리하고 `getFilteredTodos()`에서 필터링하는 구조를 먼저 잡은 뒤, 더 나은 방식이 있는지 AI에게 물어봤다. AI는 탭마다 별도 렌더 함수를 두는 방식을 제안했지만, 코드 중복이 많아진다고 판단해 채택하지 않았다.
- 직접 수정한 부분 : 탭마다 함수를 따로 두는 대신 `currentFilter` 상태 변수 하나로 통합하고, `getFilteredTodos()` 함수에서 한 번에 필터링하도록 직접 작성했다.
- 수정 이유 : 필터 조건이 바뀌어도 한 곳(`getFilteredTodos`)만 수정하면 되는 구조가 유지보수에 유리하다고 판단했다.

### Todo 일간 뷰 구현하기
- AI 활용 내용 : Todo에 `date` 필드를 추가하고 `currentDate`와 비교해 필터링하는 방향을 먼저 설계했다. 날짜 이동 로직(`addDays`)을 직접 작성하다가 `Date` 객체의 UTC/로컬 시간 차이로 날짜가 하루 밀리는 문제가 생겨서, 원인과 해결 방법을 AI에게 물어봤다.
- 직접 수정한 부분 : `new Date(dateStr)`를 `new Date(dateStr + 'T00:00:00')`로 수정해 로컬 시간 기준으로 파싱하도록 고쳤다.
- 수정 이유 : `new Date('2026-06-03')`은 UTC 자정으로 파싱되어 한국(UTC+9) 기준으로는 전날 오후 9시가 되는 문제가 있었다. 원인을 이해한 뒤 직접 수정했다.

### 로컬스토리지 연동하기
- AI 활용 내용 : `saveToStorage` / `loadFromStorage` 함수를 직접 작성한 뒤, 예외 상황(데이터 손상, 파싱 실패)에서 어떻게 처리하는 게 좋을지 AI에게 물어봤다.
- 직접 수정한 부분 : `loadFromStorage()`에 `try/catch`를 추가해 JSON 파싱 실패 시 빈 배열로 폴백하도록 했다.
- 수정 이유 : localStorage 데이터가 손상됐을 때 앱 전체가 멈추는 상황을 막기 위한 방어 처리다. AI가 엣지 케이스로 짚어준 부분을 직접 코드에 반영했다.

## 구현하면서 고민한 점

- 고민한 점 : 수정 버튼을 눌렀을 때 DOM을 직접 바꾸는 방식과, 상태 변수로 관리한 뒤 `render()`를 다시 호출하는 방식 중 어느 쪽이 좋을지 고민했다.
- 해결 방법 : DOM 직접 조작 방식은 여러 항목이 동시에 수정 모드가 되는 버그가 생길 수 있어서, `editingId` 상태 변수를 두고 `render()` 안에서 해당 항목만 입력 폼으로 그리는 방식을 선택했다. 상태 변경 → `render()` 흐름을 일관되게 유지하면 버그 추적이 훨씬 쉬워진다는 걸 느꼈다.

- 고민한 점 : JavaScript의 `Date` 객체로 날짜를 파싱할 때 예상과 다르게 하루가 밀리는 문제가 생겼다.
- 해결 방법 : `new Date('2026-06-03')`는 UTC 자정으로 파싱되어 한국(UTC+9) 기준으로는 전날 오후 9시가 된다. `'T00:00:00'`을 붙여 로컬 시간 자정으로 파싱하면 해결되는 것을 확인했다.

## 과제 회고

- 잘한 점 : 상태 변경 → `saveToStorage()` → `render()` 라는 단순한 흐름을 일관되게 유지했다. 덕분에 기능을 추가할 때마다 어디를 수정해야 하는지 명확했고, 버그가 생겼을 때도 원인을 찾기 쉬웠다.
- 아쉬운 점 : `render()`가 상태가 조금만 바뀌어도 목록 전체를 다시 그려서 비효율적이다. 나중에 React를 배우고 나면 왜 가상 DOM이 필요한지 이해가 될 것 같다.
- 다음에 시도해볼 것 : 도전 미션인 주간 뷰를 추가해보고 싶다. 또 드래그로 순서를 바꾸는 기능이 있으면 실제 투두 앱처럼 쓸 수 있을 것 같아서 시도해보고 싶다.
